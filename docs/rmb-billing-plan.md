# 人民币余额计费系统方案

## Summary

把当前“充值余额”扩展成正式的人民币计费系统，账户单位统一显示为 `元`，充值仍走现有君徕支付。生成费用规则固定为：

- 图片：按输出张数收费，`0.15 元 / 张`
- 视频：按请求 `duration` 收费，`1.5 元 / 秒`

扣费时机采用“成功后扣费”：

- 图片：上游生成成功后再扣费
- 视频：按请求时长定价，但只在任务进入成功态后扣费
- 余额不足时不允许透支；生成前做余额预检，生成成功后再做原子扣费校验

## Key Changes

### 账户与账务模型

- 保留现有 `accounts.json` 作为用户余额主存储，但账户单位统一为 `CNY`，前端全部按“元”展示，不再出现“点”“积分”等文案。
- 新增独立账务流水存储，建议新增 `billing-events.json`，记录所有资金变动：
  - `type`: `recharge_paid` / `image_generation_charge` / `video_generation_charge` / `admin_adjustment`
  - `userId`
  - `amountDelta`
  - `currency`
  - `balanceBefore`
  - `balanceAfter`
  - `referenceType`
  - `referenceId`
  - `metadata`
  - `createdAt`
- 充值成功时仍增加余额，但同时写一条 `recharge_paid` 流水；图片/视频成功扣费时写消费流水；管理员调账也写流水。

### 计费与扣费规则

- 图片计费：
  - 价格公式：`imageCharge = 0.15 * outputCount`
  - `outputCount` 取请求里的 `n`；未传时按 `1`
  - 服务端对图片生成在调用上游前先做余额预检：余额必须 `>= imageCharge`
  - 上游成功后，在持久化历史前执行原子扣费；扣费成功才写历史并返回成功
- 视频计费：
  - 价格公式：`videoCharge = 1.5 * requestedDurationSeconds`
  - 计费秒数直接取请求 `duration`
  - 先做余额预检：余额必须 `>= videoCharge`
  - 任务创建成功时只记录“预计费用”，不立即扣费
  - 等轮询更新或回调把任务状态推进到成功态时，再做原子扣费并写消费流水
  - 失败、取消、超时任务不扣费
- 不允许负余额。
- 原子扣费失败处理：
  - 如果预检通过但最终扣费时余额已不足，图片请求返回明确余额不足错误且不写入历史；视频任务标记为 `billing_failed` 或等价状态，不把成功结果暴露为可用任务。

### 后端接口与行为变化

- 保持现有接口路径不变：
  - `GET /api/account`
  - `POST /api/recharge/orders`
  - `POST /api/images/generations`
  - `POST /api/videos/generations`
- `GET /api/account` 返回值语义调整为人民币余额：
  - `balance` 继续用数字
  - `currency` 固定为 `CNY`
  - `stats` 保持不变
- 建议新增只读接口：
  - `GET /api/billing/events`
  用于前端展示充值和消费流水
- 图片生成响应建议补充账务字段：
  - `billing.amount`
  - `billing.currency`
  - `billing.type=image_generation_charge`
- 视频任务详情/列表建议补充：
  - `estimatedCharge`
  - `chargedAmount`
  - `billingStatus`，至少支持 `pending` / `charged` / `billing_failed`
- 充值接口保持按人民币金额创建订单，不引入“积分兑换率”。

### 前端与产品文案

- 首页、个人中心、账单中心全部改成“余额（元）”语义：
  - 去掉“点”“积分”等文案
  - 账户概览直接显示人民币余额
- 账单中心增加明确的价格说明：
  - `图片生成：0.15 元/张`
  - `视频生成：1.5 元/秒`
- 图片生成页在提交前展示本次费用预估：
  - 例如 `本次预计扣费：0.60 元`
- 视频生成页在提交前展示：
  - `4s 视频预计扣费：6.00 元`
- 余额不足时，前端收到后端错误后统一提示并引导去 `/recharge`
- 如果新增 `GET /api/billing/events`，账单中心展示最近充值与消费流水，区分“充值”“图片消费”“视频消费”。

### 实现策略

- 新增统一计费模块，集中定义价格常量和金额计算，避免在图片/视频服务里各自写死。
- 金额内部统一使用“分”做精确存储和运算：
  - 图片单价 `15` 分
  - 视频单价 `150` 分/秒
  - 账户余额、订单金额、流水金额全部按分落库
  - 前端展示时再格式化为元
- 现有充值订单已经按分工作，直接沿用；账户接口展示时继续走格式化显示。
- `DataStore` 需要补两个原子能力：
  - 追加账务流水
  - 原子扣款并返回扣款前后余额
- 图片成功扣费、视频成功扣费、充值入账都通过同一账务写路径完成，避免余额和流水不一致。

## Test Plan

- 账户展示：
  - `GET /api/account` 返回 `currency=CNY`
  - 首页/个人中心/账单中心均显示人民币余额
- 图片计费：
  - `n=1` 成功后扣 `0.15`
  - `n=4` 成功后扣 `0.60`
  - 余额不足时直接拒绝请求
  - 上游失败时不扣费、不写消费流水
- 视频计费：
  - `duration=4` 成功任务最终扣 `6.00`
  - `duration=8` 成功任务最终扣 `12.00`
  - 任务失败/取消/未完成不扣费
  - 余额不足时不允许创建任务
- 充值：
  - 支付回调成功后余额增加
  - 同时生成 `recharge_paid` 流水
- 并发与一致性：
  - 两个并发生成请求都通过预检时，最终只能有余额足够的请求扣费成功
  - 不出现负余额
  - 不出现“余额变了但没有流水”或“有流水但余额没变”
- 回归：
  - 现有登录、管理员后台、充值下单、图片生成、视频任务、支付回调 smoke 继续通过
  - 管理员手工调账继续可用，并写账务流水

## Assumptions

- 账户对用户展示为“人民币余额”，不是积分点数。
- 图片价格固定为 `0.15 元/张`，按请求 `n` 计费。
- 视频价格固定为 `1.5 元/秒`，按请求 `duration` 计费，不按上游最终实际秒数修正。
- 扣费时机采用“成功后扣费”，但仍做生成前余额预检。
- 不允许透支，不支持负余额。
- 现阶段仍基于本地 JSON 存储，不引入数据库迁移。
